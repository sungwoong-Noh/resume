## 프로젝트 목적

국가기관의 자원 계획·관리를 체계화하기 위한 시스템을 0에서 구축하는 프로젝트였습니다.
외부 시스템(nTOPS)의 수만 건 자산 데이터를 주기적으로 동기화하고, 담당자가 자산 계획·조회·보고를 원스톱으로 처리할 수 있는 플랫폼을 설계·개발·운영하는 것이 목표였습니다.

공공기관 특성상 서비스 중단이 허용되지 않는 환경이었고, 성능 문제가 업무 전체 흐름에 직접 영향을 미치는 구조였습니다. 이 프로젝트를 통해 **"보이지 않는 문제를 보이게 만드는 것"**, 즉 관측성(Observability)의 중요성을 가장 깊이 체감한 경험이었습니다.

## 핵심 문제 해결 과정

### 문제 1. 데이터 동기화 2시간 → 10초 (99% 개선)

**상황**

외부 시스템(nTOPS)의 수만 건 자산 데이터를 주기적으로 동기화해야 했습니다.
초기 구현에서는 동기화 1회에 **2시간**이 소요되어, 담당자가 최신 데이터를 확인하려면 다음 날까지 기다려야 했습니다.
더 심각한 문제는 **문제가 발생하고 있다는 사실 자체를 파악하기 어려웠다는 것**입니다. 동기화가 느리게 도는 동안 어느 단계에서 시간이 소비되는지 알 방법이 없었습니다.

**원인 추적 과정 (관측에서 시작)**

1. Kubernetes Pod 로그를 분석하여 동기화 함수가 시작 후 오랜 시간 동안 INSERT 구간에서 머무는 것을 확인
2. DB 쿼리 실행 로그(`show variables like 'general_log'` 활성화)로 실제 실행되는 쿼리 수를 측정 → **5만 건 데이터에 5만 번 단건 INSERT 실행** 확인
3. Hibernate 배치 처리 문서를 분석하여 `IDENTITY` 전략과 Batch Insert의 비호환성 발견

**해결 과정**

1. Hibernate `jdbc.batch_size` 설정 적용
2. ID 생성 전략 `IDENTITY` → `TABLE` 전략으로 교체 후 배치 동작 확인
3. 청크 단위(1,000건) 분할로 메모리 OOM 방지 및 트랜잭션 범위 최소화

**결과:** 동기화 소요 시간 **2시간 → 10초 (99.9% 개선)**

**이 경험이 가르쳐 준 것**

로그와 메트릭 없이는 "느리다"는 증상만 알 뿐, 어디가 문제인지 알 수 없었습니다. 쿼리 실행 로그를 활성화하는 단순한 조치가 5만 번의 INSERT라는 명확한 원인을 드러냈습니다. **관측 가능한 시스템이 곧 고칠 수 있는 시스템**이라는 것을 체감한 전환점이었습니다.

### 문제 2. 핵심 조회 p95 17초 → 3초 (82% 개선) — 실행계획 기반 진단

**상황**

국정자원 통합 운영 전환 후(2025.07), 핵심 자원 조회 화면의 응답이 간헐적으로 매우 느려지는 민원이 접수되었습니다.
"가끔 느리다"는 증상만 있어 재현 자체가 어려운 상황이었습니다.

**원인 추적 과정**

1. Grafana 대시보드(당시 팀에서 운영 중인 모니터링 환경)에서 해당 시간대 DB 커넥션 수·쿼리 처리 시간 급등 확인
2. 느린 쿼리 로그(`slow_query_log`) 활성화 → 특정 자원 조회 쿼리가 반복적으로 상위에 등장
3. `EXPLAIN` 으로 실행계획 분석 → 인덱스 미사용, 전체 테이블 스캔(Full Scan) 발생 확인
4. Tibero 힌트와 복합 인덱스 추가 후 실행계획 재분석으로 Index Range Scan 전환 확인

**결과:** p95 응답시간 **17초 → 3초 (82% 개선)**

**이 경험이 가르쳐 준 것**

Grafana 대시보드가 없었다면 "가끔 느리다"는 민원은 재현조차 못 했을 것입니다. 메트릭이 시간대를 특정해 주었고, slow query log가 쿼리를 특정해 주었으며, 실행계획이 인덱스를 특정해 주었습니다. **계층적인 관측 도구가 문제를 단계적으로 좁혀주는 과정**이 인상적이었고, 이것이 이후 제가 직접 모니터링 체계를 구축하게 된 동기가 되었습니다.

---

### 이 프로젝트 이후: 모니터링 체계를 직접 구축하다

RPMS 운영 과정에서 "로그와 메트릭이 있을 때와 없을 때의 차이"를 몸으로 경험한 뒤, 현재 MSP 환경(플랜아이)에서 고객사 인프라 모니터링 체계를 직접 설계·구축하고 있습니다.

- **Prometheus + Grafana** 기반 기관별 모니터링 서버 구축
- **VictoriaMetrics** 도입으로 메트릭 보존 기간 **30일 → 1년** 확장
- **Ansible** 기반 Exporter 일괄 설치 자동화

와탭랩스에서 모니터링 서비스를 개발하는 역할에 지원한 이유가 바로 여기 있습니다. 저는 모니터링 도구를 사용하는 입장에서 "무엇이 현장에 필요한가"를 실제로 경험했고, 이제 그 도구를 만드는 쪽에서 기여하고 싶습니다.

## 회고

이 프로젝트는 저에게 **"좋은 백엔드 개발자는 시스템을 안에서만 보는 사람이 아니다"** 라는 것을 가르쳐 주었습니다.

성능 문제를 해결할 때마다 결정적인 단서는 항상 코드 밖에 있었습니다. 쿼리 실행 로그, 메트릭 그래프, 실행계획 — 이 관측 데이터들이 없었다면 저는 코드만 뒤지며 추측에 의존했을 것입니다.

와탭랩스가 만드는 APM·모니터링 서비스는 저처럼 시스템 내부에서 문제를 해결하려는 개발자들에게 가장 필요한 도구입니다. 그 도구를 사용하는 입장에서 느꼈던 필요와 한계를, 이제 도구를 만드는 쪽에서 반영하고 싶습니다.
