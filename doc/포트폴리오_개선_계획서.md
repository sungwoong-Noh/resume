## 1. 포트폴리오 ↔ 이력서 톤 통일

### 현황

| | 이력서 (`/`) | 포트폴리오 (`/portfolio`) |
|-|------------|------------------------|
| 배경 | `bg-gray-900` 다크 | `#f5f0e8` 크림 |
| 강조색 | `teal-400 / teal-500` | `#2a6041` 포레스트 그린 |
| 카드 | `bg-gray-800 border-gray-700` | `bg-white border-[#d6e8dc]` |
| 폰트 | Inter, mono 혼용 | Inter + Playfair Display |

### 방향

포트폴리오가 PDF를 기반으로 제작되어 이력서와 톤이 완전히 다름.  
**포트폴리오를 이력서 다크 테마로 통일**하는 방향이 적합.

- 이유: 두 페이지를 같은 도메인에서 함께 제출하는 경우, 일관된 브랜딩이 신뢰감을 줌
- PDF 포트폴리오(beige 스타일)는 별도로 유지하면 되므로 웹 버전만 통일

### 작업 내용

**변경 대상 파일**: 모든 `components/portfolio/*.tsx`

```
현재 → 변경
bg-[#f5f0e8]            → bg-gray-900
bg-[#eee8da]            → bg-gray-950 또는 bg-gray-900
bg-white                → bg-gray-800
border-[#d6e8dc]        → border-gray-700
text-[#2a6041]          → text-teal-400
bg-[#2a6041]            → bg-teal-500
font-serif (Playfair)   → font-sans (Inter)
text-gray-600/500       → text-gray-300/400
```

**커버 헤더("Career Overview", "Project Detail")**: teal gradient 또는 단순 uppercase mono 라벨로 변경  
**Career Overview 타임라인 점·선**: `bg-teal-500` / `border-gray-700`  
**Tech Stack 태그**: 이력서 스타일 `bg-gray-700 text-gray-300 font-mono` 로 통일

## 3. 이력서 ↔ 포트폴리오 설명 일관성

### 현황 문제

`data/resume.ts`와 `data/portfolio.ts`에 같은 프로젝트 설명이 각각 별도로 작성되어 있어,  
두 파일 간 수치·표현이 달라질 위험이 있음.

| 항목 | resume.ts | portfolio.ts | 불일치 예시 |
|------|----------|-------------|-----------|
| RPMS N+1 개선 | `99% 단축 (2시간 → 10초)` | `90% 이상 향상` | 수치 다름 |
| PPP Cloud 코드 재사용성 | `60% 향상` | `60% 향상` | 일치 ✓ |
| Tibero 성능 | `p95 17초 → 3초 (82% 개선)` | `핵심 조회 성능 82% 향상` | 수치 일치, 표현 다름 |

### 개선 방향

`data/resume.ts`의 `result` 문구를 **단일 진실 소스**로 정의하고,  
`data/portfolio.ts`의 `keyResponsibilities`와 `overviewContributions`는 그것을 기반으로 확장 작성.

### 작업 내용

1. resume.ts의 각 프로젝트 `result` 수치를 기준값으로 확정
2. portfolio.ts에서 동일 프로젝트의 해당 수치를 resume.ts 기준으로 교정
3. 이후 수치 변경 시 resume.ts → portfolio.ts 순서로 동기 수정

## 4. 포트폴리오 PAR 구조 상세화

### 현황

포트폴리오의 `keyResponsibilities`는 단순 bullet 결과 나열.  
이력서의 `PARItem { result, problem?, analyze? }` 구조가 포트폴리오에는 미적용.

### 방향

포트폴리오는 이력서보다 **더 여유 있는 공간**을 가지므로,  
P(문제 배경) → A(선택 이유) → R(결과) 흐름을 항목별로 블록으로 표시.

### 타입 변경 계획

```ts
// 현재
keyResponsibilities: string[]

// 변경
type PortfolioPARItem = {
  result: string        // 헤드라인
  problem?: string      // 왜 이 문제가 중요했나
  analyze?: string      // 왜 이 방법을 선택했나
}

keyResponsibilities: PortfolioPARItem[]
```

### 렌더링 방향 (`ProjectDetail.tsx`)

이력서의 토글 없이 항상 펼친 방식(현재 이력서 PARAccordion 참고)을 포트폴리오에서는 더 크게 표시:

```
→ [result 헤드라인]
  ┌─────────────────────────────
  │ 문제  [problem 텍스트]
  │ 분석  [analyze 텍스트]
  └─────────────────────────────
```

### 적용 대상 핵심 항목 (PAR 필요)

| 프로젝트 | 항목 | P | A |
|---------|------|---|---|
| PPP Cloud | Saga 패턴 | 장비 API ↔ Spring 트랜잭션 분리 | 2PC 불가 → Saga 선택 |
| PPP Cloud | Self-Service API | 수동 처리 월 50건 | Managed 방식의 구조적 한계 |
| RPMS | N+1 + Bulk Insert | 14만건 2시간 배치 | JPA IDENTITY 전략 단건 INSERT 강제 한계 |
| RPMS | QueryDSL 도입 | Native 쿼리 유지보수 불가 | 파일럿 적용 후 팀 설득 |
| 국정자원 운영 | Tibero 최적화 | 핵심 조회 p95 17초 | 타입 불일치가 인덱스 무력화 근본 원인 |
| 통합 커머스 | Rate Limiting | 다운스트림 보호 필요 | 토큰 버킷 선택 이유 |

## 5. 애매한 프로젝트 제거

### 제거 대상

| 프로젝트 | 제거 이유 |
|---------|---------|
| **충남도청 SMS 서비스 유지보수** | 노후 서버 교체 위주의 인프라 작업. 백엔드 개발 역량보다 운영/인프라에 가까워 포트폴리오 주제와 결이 맞지 않음. 기간도 3개월로 짧음 |
| **과학문화체험 플랫폼** | eGovFramework 기반 단기(3개월) 프로젝트. 현재 기술 스택(Spring Boot MSA)과 동떨어져 있고, Soft Reservation 외에 포트폴리오에서 어필할 기술적 깊이가 부족 |

> **단, 이력서(`data/resume.ts`)에서는 제거하지 않음.** 경력 기간 증빙이 필요하므로 이력서에는 유지.

### 제거 후 남는 프로젝트 (5개)

1. **PPP Cloud** — Saga 패턴, Self-Service API 자동화
2. **통합 커머스 프로젝트** — AWS ECS MSA, Rate Limiting
3. **국정자원 시스템 통합운영** — Tibero 성능 최적화, 운영 체계화
4. **RPMS** — QueryDSL 도입, N+1 + Bulk Insert
5. **한전 자율안전 솔루션** — GIS 백엔드, CI/CD 구축 (커리어 초기 임팩트)
