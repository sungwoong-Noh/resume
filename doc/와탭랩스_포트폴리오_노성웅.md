## 와탭랩스에 지원하는 이유

지금 하는 일은 고객사 인프라 모니터링 체계를 직접 설계하고 구축하는 것입니다. Prometheus, Grafana, VictoriaMetrics를 운영하다 보면 쓰기만 할 때는 보이지 않던 것들이 있습니다. 수집 주기를 촘촘하게 가져가면 저장 비용이 따라 올라가고, 레이블을 함부로 늘리면 카디널리티가 터집니다. 알림 규칙이 많아지면 정작 중요한 것을 놓치기도 합니다.

그 전에는 이 도구들을 쓰는 입장이었습니다. 쿼리 실행 로그로 5만 번 단건 INSERT를 찾아냈고, Grafana 대시보드로 간헐적인 느린 응답의 시간대를 특정했습니다. 도구가 보여주는 대로 봤고, 그게 문제를 해결해줬습니다.

두 입장을 모두 경험하고 나니 자연스럽게 생각이 생겼습니다. 현장에서 실제로 쓸 수 있는 APM은 어떻게 설계돼야 하는가. 그 질문을 가지고 와탭랩스에서 일하고 싶습니다.

## 프로젝트 1. RPMS — 관측 데이터로 추적한 성능 최적화

**기간:** 2024.01 ~ 2025.06 | **역할:** 백엔드 리드 개발자
**Tech Stack:** Spring Boot · JPA · QueryDSL · MariaDB · Redis · Kubernetes · Jenkins · Helm

### 프로젝트 개요

국가기관의 자원 계획·관리 시스템을 0에서 설계·개발·운영한 프로젝트입니다. 외부 시스템(nTOPS)의 수만 건 자산 데이터를 동기화하고, 담당자가 계획·조회·보고를 원스톱으로 처리할 수 있는 플랫폼을 End-to-End로 담당했습니다.

---

### 문제 해결 1. 데이터 동기화 2시간 → 10초 (99% 개선)

#### 관측에서 시작한 원인 추적

동기화가 느리다는 증상만 있을 뿐, **어느 단계에서 시간이 소비되는지 알 방법이 없었습니다.**

1. Kubernetes Pod 로그로 동기화 함수가 INSERT 구간에서 오랜 시간 머무는 것을 확인
2. DB 쿼리 실행 로그(`general_log`) 활성화 → 5만 건 저장에 5만 번 단건 INSERT 실행 확인
3. Hibernate 배치 처리 문서 분석 → `IDENTITY` 전략과 Batch Insert 비호환성 발견

#### 해결 과정

```
문제: IDENTITY 전략 → Hibernate가 ID를 즉시 알아야 → 배치 비활성화
해결: TABLE 전략으로 교체 → Hibernate Batch Insert 동작 확인
추가: 청크 단위(1,000건) 분할로 OOM 방지 + 트랜잭션 범위 최소화
```

**결과:** 동기화 2시간 → 10초 (99.9% 개선)

> 이 경험에서 배운 것: 로그가 없었다면 "느리다"는 증상만 알 뿐 원인을 알 수 없었습니다. `general_log` 한 줄이 5만 번의 INSERT라는 명확한 원인을 드러냈습니다.

---

### 문제 해결 2. 핵심 조회 p95 17초 → 3초 (82% 개선)

#### 계층적 관측으로 좁혀간 원인

"가끔 느리다"는 민원이 접수됐지만 재현이 어려운 상황이었습니다.

```
1단계: Grafana 대시보드 → 해당 시간대 DB 커넥션 수 급등 확인 (시간대 특정)
2단계: slow_query_log 활성화 → 특정 자원 조회 쿼리 반복 등장 (쿼리 특정)
3단계: EXPLAIN 실행계획 → Full Scan 발생 (원인 특정)
4단계: 복합 인덱스 + Tibero 힌트 적용 → Index Range Scan 전환 확인
```

**결과:** p95 17초 → 3초 (82% 개선)

> Grafana가 없었다면 재현조차 못 했을 것입니다. 메트릭 → 쿼리 → 실행계획 순으로 계층적으로 좁혀가는 과정이 와탭랩스가 만드는 APM의 가치를 직접 체감한 순간이었습니다.

---

### 문제 해결 3. QueryDSL 도입으로 런타임 오류 제거

기존 Native SQL 문자열 관리로 컬럼명 변경 시 런타임에서만 오류가 발견되는 상황이 반복되었습니다.

- 팀 내 최초 QueryDSL 도입 제안 → POC → 팀 설득 → 전사 적용
- Native 쿼리 70% 이상 QueryDSL로 전환
- BooleanBuilder 패턴으로 15개 필터 조합 조회 단순화
- 컨벤션 문서화 → 코드 리뷰 체크리스트 반영

**결과:** 런타임 SQL 오류 0건, 유지보수 비용 50% 절감

## 프로젝트 2. 통합 커머스 프로젝트 — MSA 환경 관측성 설계

**기간:** 2025.07 ~ 2025.09 | **역할:** 인프라·인증 파트 담당
**Tech Stack:** Spring Boot · Spring Cloud · JPA · PostgreSQL · AWS ECS · GitHub Actions · Zipkin · Prometheus · Grafana

### 프로젝트 개요

패스트캠퍼스 InnerCircle 부트캠프를 통해 진행된 통합 커머스 프로젝트입니다. 상품·주문·결제·회원 도메인을 MSA 아키텍처로 설계하고, 대규모 트래픽 환경에서도 안정적인 서비스를 목표로 했습니다.

---

### 관측성 설계: 분산 시스템에서 "무슨 일이 일어나는지" 보이게 만들기

MSA는 서비스 간 호출이 복잡하게 얽혀 있어, 장애 발생 시 어느 서비스에서 문제가 생겼는지 파악하기 어렵습니다.

#### Zipkin 분산 추적 도입

```
목적: 마이크로서비스 간 호출 흐름 가시화
방법: B3 전파 방식으로 Trace ID를 서비스 간 전달
효과: 병목/오류 지점을 Trace 단위로 식별
```

- 주문 요청 → 상품 서비스 → 결제 서비스 전체 흐름을 단일 Trace로 추적
- 특정 서비스에서 응답 지연 발생 시 Zipkin UI에서 즉시 식별 가능
- 장애 원인 진단 시간 단축

#### Prometheus + Grafana 모니터링 구성

- 각 서비스별 메트릭 수집 (요청 수, 응답 시간, 에러율)
- Grafana 대시보드로 서비스 상태 실시간 모니터링
- Rate Limiting 적용 후 트래픽 패턴 변화 대시보드에서 확인

---

### 인프라: AWS ECS 기반 Auto-scaling MSA

```
AWS ECS + Auto-scaling: 트래픽 급증 시 자동 확장
AWS CloudMap: Service Discovery 기반 서비스 간 통신
API Gateway: 중앙 인증 + Rate Limiting + 라우팅
GitHub Actions + ECR: CI/CD 자동화 (비용 100% 절감)
```

#### JWT 무상태 인증 체계

- API Gateway 중앙 인증으로 서비스 간 인증 의존성 제거
- 인증 완료 사용자 정보를 `X-User-ID` 커스텀 헤더로 전달
- 서비스 간 결합도 완화 + 수평 확장 용이한 구조

#### 토큰 버킷 Rate Limiting

- API Gateway 레벨에서 스파이크 트래픽 흡수
- 다운스트림 서비스 보호 및 시스템 안정성 확보

## 프로젝트 3. MSP 모니터링 체계 구축 — 모니터링을 직접 만들다

**기간:** 2026.02 ~ 현재 | **역할:** System Engineer (플랜아이)
**Tech Stack:** Prometheus · Grafana · VictoriaMetrics · Ansible · KT Cloud

### 프로젝트 개요

MSP(Managed Service Provider) 사업부에서 KT Cloud 기반 고객사 인프라의 모니터링 체계를 설계·구축하고 있습니다. 와탭랩스가 제공하는 것과 유사한 서비스를 **직접 구축하는 경험**입니다.

---

### 아키텍처 설계 및 구축

#### Prometheus + Grafana 기관별 모니터링 서버 구축

```
구성:
- 기관별 독립 Prometheus 서버 (멀티테넌트 격리)
- Grafana 대시보드 (서버 자원, 네트워크, 프로세스 모니터링)
- CMP 개발팀과 보고서 자동 생성 기능 연계
```

#### VictoriaMetrics 도입 — 메트릭 장기 보존 문제 해결

Prometheus의 기본 설정으로는 메트릭 보존 기간이 30일로 제한되어, 고객사가 장기 트렌드 분석을 할 수 없었습니다.

```
문제: Prometheus 기본 보존 기간 30일 → 장기 추세 분석 불가
해결: VictoriaMetrics를 Remote Storage로 도입
결과: 메트릭 보존 기간 30일 → 1년 확장
추가 효과: 높은 카디널리티 메트릭 처리 성능 향상, 압축률 개선
```

#### Ansible 기반 Exporter 자동화

- 수십 개 서버에 Exporter를 수동 설치하는 반복 작업을 Ansible Playbook으로 자동화
- Node Exporter, Process Exporter, Custom Exporter 일괄 배포
- 시스템 자원 조회 자동화로 운영 효율화

---

### 이 경험이 준 시각

모니터링 체계를 직접 구축하면서 **"좋은 모니터링 도구가 무엇인지"** 사용자 관점에서 명확하게 이해하게 되었습니다.

- **수집 주기와 보존 기간의 트레이드오프** — 세밀한 수집은 저장 비용을 높임
- **카디널리티 폭발 문제** — 레이블 설계가 잘못되면 메트릭 수가 기하급수적으로 늘어남
- **알림 피로도** — 너무 많은 알림은 중요한 알림을 놓치게 만듦
- **대시보드 설계** — 운영자가 30초 안에 상태를 파악할 수 있어야 함

와탭랩스에서 모니터링 서비스를 개발할 때, 이 현장 경험이 실질적인 가치를 가져다 줄 것이라 생각합니다.
