## 프로젝트 1. RPMS — 0에서 시작한 시스템, 깊이 있는 문제 해결

**기간:** 2024.01 ~ 2025.06 | **역할:** 백엔드 리드 개발자
**Tech Stack:** Spring Boot · JPA · QueryDSL · MariaDB · Redis · Kubernetes · Jenkins · Helm

### 프로젝트 개요

국가기관의 자원 계획·관리 시스템을 처음부터 설계·개발·운영한 프로젝트입니다. 요구사항 정제부터 도메인 설계, API 개발, 배포, 운영까지 End-to-End를 주도했습니다. 외부 시스템(nTOPS)의 수만 건 자산 데이터를 동기화하고, 담당자가 계획·조회·보고를 원스톱으로 처리할 수 있는 플랫폼입니다.

---

### 문제 해결 1. 데이터 동기화 2시간 → 10초 (99% 개선)

동기화 1회에 2시간이 걸려 담당자가 최신 데이터를 확인하려면 다음 날까지 기다려야 했습니다.

**원인 추적**

1. Kubernetes Pod 로그 → INSERT 구간에서 지연 확인
2. DB 쿼리 실행 로그(`general_log`) 활성화 → 5만 건에 5만 번 단건 INSERT 확인
3. Hibernate 문서 분석 → `IDENTITY` 전략과 Batch Insert 비호환성 발견

**해결**

- ID 생성 전략 `IDENTITY` → `TABLE` 전략으로 교체
- `jdbc.batch_size` 설정으로 Hibernate Batch Insert 활성화
- 청크 단위(1,000건) 분할로 OOM 방지 + 트랜잭션 범위 최소화

**결과:** 동기화 2시간 → 10초 (99.9% 개선), 담당자 업무 흐름 즉시 개선

---

### 문제 해결 2. 핵심 조회 p95 17초 → 3초 (82% 개선)

"가끔 느리다"는 민원이 접수됐지만 재현이 어려운 상황이었습니다.

**원인 추적**

1. Grafana 대시보드 → 해당 시간대 DB 커넥션 수 급등으로 시간대 특정
2. `slow_query_log` 활성화 → 특정 자원 조회 쿼리 반복 등장
3. `EXPLAIN` 실행계획 → Full Scan 발생 확인
4. 복합 인덱스 + Tibero 힌트 적용 → Index Range Scan 전환

**결과:** p95 17초 → 3초 (82% 개선)

---

### 문제 해결 3. QueryDSL 도입으로 유지보수 비용 절감

기존 Native SQL 문자열 관리로 컬럼명 변경 시 런타임에서만 오류가 발견되는 상황이 반복되었습니다.

- 팀 내 최초 QueryDSL 도입 제안 → POC → 팀 설득 → 전사 적용
- Native 쿼리 70% 이상 QueryDSL로 전환
- BooleanBuilder 패턴으로 15개 필터 조합 조회 단순화

**결과:** 런타임 SQL 오류 0건, 유지보수 비용 50% 절감

## 프로젝트 2. PPP Cloud — 사용자 워크로드 90% 절감, 클라우드 셀프서비스 플랫폼

**기간:** 2024.08 ~ 2025.06 | **역할:** 백엔드 개발자
**Tech Stack:** Spring Boot · JPA · PostgreSQL · Redis · KT Cloud API · Saga 패턴 · Kubernetes · Jenkins

### 프로젝트 개요

KT Cloud와 협력하여 개발한 클라우드 자원 셀프서비스 플랫폼입니다. 기존에는 클라우드 자원 신청·변경·반납을 담당자가 수동으로 처리했는데, 이를 사용자가 직접 처리할 수 있도록 자동화했습니다.

---

### 핵심 문제 해결 1. Self-Service API — 운영 워크로드 90% 절감

클라우드 자원 신청이 들어오면 담당자가 KT Cloud 콘솔에 직접 접속해 수동으로 처리하는 구조였습니다. 신청 건수가 늘어날수록 처리 지연과 휴먼 에러가 반복되었습니다.

**해결 과정**

- KT Cloud API 분석 → 자원 생성·변경·삭제 자동화 API 설계
- 신청 → 승인 → 프로비저닝 → 완료 전 과정을 시스템이 처리하도록 구현
- 예외 상황(API 타임아웃, 자원 생성 실패) 처리 로직 설계

**결과:** 담당자 수동 처리 워크로드 90% 절감, 처리 시간 단축

---

### 핵심 문제 해결 2. Saga 패턴으로 분산 트랜잭션 일관성 확보

클라우드 자원 프로비저닝은 여러 외부 API를 순서대로 호출하는 구조라, 중간에 실패하면 자원이 부분적으로만 생성되는 문제가 있었습니다.

**해결 과정**

- Choreography 기반 Saga 패턴 도입
- 각 단계별 보상 트랜잭션(Compensating Transaction) 설계
- 실패 시 롤백 흐름 자동화로 데이터 일관성 확보

**결과:** 부분 생성 장애 0건, 운영 신뢰도 향상

---

### 핵심 문제 해결 3. 공통 모듈화로 중복 코드 60% 제거

서비스별로 유사한 인증·로깅·에러 처리 코드가 중복 관리되어 변경 시 누락이 반복되었습니다.

- 공통 모듈 설계 및 분리
- 각 서비스에서 의존성으로 주입하는 구조로 전환
- 중복 코드 60% 이상 제거, 일관된 에러 응답 구조 확립
